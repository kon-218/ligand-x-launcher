# Manual Windows launcher build (CI-bypass)

The release CI (`.github/workflows/launcher-release.yml`) historically failed to
produce a Windows installer because `makensis` was not on the build step's PATH
(choco updates the *machine* PATH, but the running step's PATH is stale). The fix
is committed as `e280340` but requires a **`workflow`-scoped** token to push.

Until that lands, the Windows launcher (`ligandx-windows-amd64.exe`) is built
**locally from Linux** and uploaded directly to the `latest` release. This works
because the public launcher's frontend is fully static and pre-embedded
(`frontend-public/` + committed `wailsjs/`, `frontend:build` empty in
`wails.json`), and Wails uses a pure-Go WebView2 loader — so **no CGO and no
`wails build` are required**; a plain `go build` with the embed suffices.

## Reproduce

```bash
# 1. Go toolchain (any 1.22+; go.mod toolchain directive auto-fetches as needed)
export PATH=$PATH:/path/to/go/bin GOPATH=/tmp/gopath GOCACHE=/tmp/gocache

cd ligand-x-launcher

# 2. (optional polish) embed manifest + icon + version info, like wails does.
#    Manifest sets asInvoker (NO admin), PerMonitorV2 DPI, Common-Controls v6.
go install github.com/josephspurrier/goversioninfo/cmd/goversioninfo@latest
#    versioninfo.json -> IconPath: build/windows/icon.ico, ManifestPath: ligandx.exe.manifest
goversioninfo -64 -o resource_windows_amd64.syso versioninfo.json

# 3. Cross-compile the public launcher (GUI subsystem, no console window)
GOOS=windows GOARCH=amd64 CGO_ENABLED=0 go build -tags public \
  -ldflags="-H windowsgui -s -w" -o ligandx-windows-amd64.exe .

# 4. IMPORTANT: remove the .syso afterward so a future `wails build` in this dir
#    doesn't double-embed the manifest and fail to link.
rm -f resource_windows_amd64.syso

# 5. Upload to the `latest` release (repo-scoped token is enough; no workflow scope)
#    Delete the old same-named asset first, then POST to the uploads API.
```

The produced exe is a **portable** launcher (run directly, no install). Verified:
`PE32+ executable (GUI) x86-64`, manifest embedded (`asInvoker`, `dpiAware`),
anonymously downloadable from the release.

## Proper fix (preferred, when a workflow-scoped token is available)

`git push origin main` to land `e280340`; CI then builds the real NSIS installer
with `makensis` on PATH and `if-no-files-found: error` so it can never silently
ship nothing again.

## Core image / version contract

- Core images are pinned to a concrete tag (currently `v2026.06.21`), published at
  `ghcr.io/kon-218/ligand-x/*` and **public** (anonymous pull) for core services.
- `.env.production.template` ships `VERSION=v2026.06.21` (concrete, not `latest`),
  so the launcher's install-time pin leaves it untouched and Start accepts it.
- `INTERNAL_WORKER_SECRET` is auto-generated by `ensureProductionEnv` (app.go).
- Rebuild/push core images with `../push_core.sh v2026.06.21` (needs a
  `write:packages` GHCR login).
